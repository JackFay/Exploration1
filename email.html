Recap

This exploration was much trickier than I thought it would be...It turns out that IIS isn't the easiest to configure a SMTP Server with, and that it requires many workarounds to actually send an email. I discovered that there are (unsurprisingly) many security issues behind setting up a successful email server. This is due to the fact that using a function such as php's "mail()" allows one to easily change the "from" header of an email. This means that it is possible to "fake" an email and make it appear like it came from someone, whom in reality did not send it. The article about configuring an SMTP Server in IIS details some of these security issues. I personally was able to send a few emails, however, they only successfully sent to my gmail account and had warnings from gmail saying that gmail "could not verify" the sender of the mail. I also noticed that the sending of the email was very delayed. Many email companies have realized the importance of verifying a sender, so in order to prevent a warning message, I would have to file a "PTR Record" for reverse DNS lookups. A PTR Record essentially allows your server's ip address to be traced back to your hostname. Overall it was a very valuable learning experience, as I had to learn about/incorporate a variety of web technologies.
